# AGENTS.md — ISAP2

Этот файл задаёт постоянные правила работы ИИ-агентов в репозитории ISAP2.

Подробное состояние проекта, история решений и текущая точка продолжения находятся в `HANDOFF.md`. Этот файл определяет **как работать**, а `HANDOFF.md` — **с чем и от какой точки продолжать**.

## 1. Приоритет инструкций

1. Выполняй явный текущий запрос пользователя.
2. Соблюдай инструкции ближайшего применимого `AGENTS.md` или `AGENTS.override.md`.
3. Соблюдай правила корневого `AGENTS.md`.
4. Используй `HANDOFF.md` как контекст, но проверяй его утверждения по текущему коду.
5. При расхождении документации и репозитория источником истины являются код, конфигурация, миграции и свежие результаты тестов.

Не меняй архитектуру, контракт или данные только ради согласования кода с устаревшим отчётом.

## 2. Цель проекта

ISAP2 — система подготовки документов промышленной безопасности. Главный текущий приоритет — воспроизводимый путь `анкета/DOCX → preview → привязка организации/ОПО → context v2 → DOCX → PDF → Quality Review → ручная экспертиза` на реальных данных.

Не расширяй проект новыми подсистемами, пока это не требуется для текущего проверяемого результата.

## 3. Рабочий язык и стиль взаимодействия

- Общайся с пользователем на русском языке.
- Пиши отчёты коротко, конкретно и с проверяемыми фактами.
- Для локальных действий пользователя давай готовые PowerShell-команды.
- Не перегружай отчёт полными логами; показывай итог, ключевые ошибки и пути к артефактам.
- Разделяй подтверждённые факты, выводы и предположения.
- Не объявляй задачу завершённой без проверки, соответствующей риску изменений.

Пользователь предпочитает:

- **основной режим** — короткий промпт для ИИ-разработчика;
- **контрольный режим** — проверка присланных логов, diff и результатов;
- **аварийный режим** — точечный patch или ZIP, когда обычный путь заблокирован.

## 4. Обязательный старт каждой задачи

Перед изменением файлов:

1. Определи корень Git-репозитория и текущий рабочий каталог.
2. Прочитай применимые `AGENTS.md`/`AGENTS.override.md` от корня до текущего каталога.
3. Прочитай релевантные части `HANDOFF.md`.
4. Выполни:

   ```powershell
   git status --short
   git branch --show-current
   git diff --stat
   git log -5 --oneline
   ```

5. Проверь наличие пользовательских незакоммиченных изменений.
6. Найди реальные команды проекта по `README`, `pyproject.toml`, `package.json`, `docker-compose.yml` и CI workflow.
7. Для Docker сначала выполни `docker compose config --services`; не угадывай имена сервисов.
8. Сформулируй один ближайший проверяемый результат.

Если пользователь просит только аудит, объяснение или диагностику, не исправляй код без отдельного указания.

## 5. Безопасность рабочего дерева

- Считай все существующие изменения принадлежащими пользователю.
- Не удаляй, не перезаписывай и не форматируй несвязанные файлы.
- Не используй `git reset --hard`, `git clean -fd`, `git checkout --`, destructive migration rollback и аналогичные команды без явного разрешения.
- Не меняй `.env`, секреты, токены и production-конфигурацию без явной необходимости и разрешения.
- Не добавляй production dependency без обоснования и проверки существующих возможностей проекта.
- Не выполняй массовый рефакторинг в задаче точечного исправления.
- Если рабочее дерево грязное, сначала определи пересечение с задачей. При конфликте остановись и сообщи пользователю.

## 6. Использование skills

Skills обязательны, когда их область соответствует задаче.

1. Проверь доступные skills до начала специализированной работы.
2. Если пользователь явно назвал skill, используй его.
3. Перед применением прочитай полный `SKILL.md`.
4. Следуй workflow, ограничениям и проверкам skill.
5. Не заявляй об использовании skill, если была только прочитана инструкция.
6. Не используй skill ради формальности, если задача ему не соответствует.

Типичные соответствия: GitHub/CI/review skills для PR и Actions; document/PDF skills для DOCX и PDF; spreadsheet skill для справочников; image skill для титульных иллюстраций; docs/research skill для актуальной документации; file/library skill для долговременных артефактов.

Если в проекте есть собственные skills, зафиксируй их путь, назначение и условия запуска в `HANDOFF.md` после проверки.

## 7. Использование MCP

Используй MCP для подключённых источников и инструментов, а не вместо локальной проверки кода.

Перед использованием MCP:

1. Проверь project-scoped `.codex/config.toml` и доступную конфигурацию Codex.
2. Получи список серверов через `/mcp` или `codex mcp list`, если поверхность это поддерживает.
3. Проверь требуемый сервер безопасным read-only вызовом.
4. Используй MCP resource/resource template вместо web-поиска, когда это подходящий подтверждённый источник.
5. В отчёте укажи источник данных и fallback, если он применялся.

Правила:

- наличие записи конфигурации не означает, что сервер подключён;
- MCP не заменяет unit, integration и E2E тесты;
- не выполняй внешние write/send/publish действия без явного разрешения;
- не сохраняй credentials в репозитории;
- при недоступности MCP используй безопасный локальный fallback и явно укажи ограничение;
- для GitHub предпочитай подключённый GitHub-инструмент для PR/issues, но используй локальный `git` для рабочего дерева;
- для актуальной технической документации используй официальные или первичные источники.

## 8. Субагенты

Для сложной задачи с независимыми направлениями используй основной агент и до трёх субагентов. Субагенты особенно полезны для read-heavy аудита, поиска, тестов и анализа логов.

Рекомендуемое разделение для ПМЛА:

| Агент | Область |
|---|---|
| Основной | План, решения, координация, интеграция, финальные тесты |
| Субагент 1 | Import, API, preview, БД, привязка организации и ОПО |
| Субагент 2 | Generation context, v2 mapper, schema, Rules/Scenario, KG/RAG |
| Субагент 3 | Шаблон, плейсхолдеры, DOCX/PDF, visual QA, Quality Review |

Обязательные правила делегирования:

- делегируй только конкретные, ограниченные и независимые подзадачи;
- передай агенту необходимые критерии и ожидаемый формат результата;
- назначай одного владельца на изменяемый файл;
- не допускай параллельное редактирование одного файла разными агентами;
- для первичного аудита предпочитай read-only работу;
- требуй от субагента пути к файлам, факты, риски, тесты и краткую рекомендацию;
- основной агент обязан проверить выводы, разрешить противоречия и выполнить общую проверку;
- дождись всех необходимых результатов перед итоговым заключением;
- не запускай субагентов для одной маленькой локальной правки.

## 9. Архитектурные ограничения ПМЛА v2

Сохраняй текущий разделённый pipeline. Не объединяй импорт, маппинг, правила и рендеринг в один сервис.

Ожидаемые слои: API/валидация; import/preview; binding и хранение анкеты; generation context; `map_to_v2_context` и schema validation; Rules/Scenario/Narrative/KG/RAG; `PmlaTemplateRenderer`; DOCX → PDF; Quality Review/review workflow.

Соблюдай следующие инварианты:

- frontend передаёт `template_version: "v2"` для v2 pipeline;
- provenance явно показывает версию шаблона и pipeline;
- `_import_meta` хранит метаданные импорта, но не попадает в пользовательский текст ПМЛА;
- `organization_candidate` и `facility_candidate` остаются читаемыми структурами;
- входные структуры не мутируются скрыто; используй копирование там, где это предусмотрено контрактом;
- неизвестные необязательные поля не должны ломать v2 context;
- отсутствие данных порождает понятное предупреждение, а не выдуманное значение;
- неоднозначная организация или ОПО не привязываются молча;
- ОПО нельзя привязать к чужой организации;
- повторная генерация на одинаковом входе должна быть эквивалентной.

## 10. Типы ОПО и защита от смешения контента

Поддерживаемые и ранее проверявшиеся типы:

- сеть газопотребления;
- котельная;
- АГЗС — станция газозаправочная (автомобильная).

Не заменяй «АГЗС» на «АЗС».

Для каждого типа изолируй:

- оборудование;
- hazards;
- scenarios;
- required services;
- appendices;
- regulations;
- narrative и forbidden terms.

Любая утечка текста, оборудования или сценария от другого типа ОПО является блокирующим дефектом.

## 11. Knowledge Graph, RAG и Graphiti/Graphify harness

В проекте ранее использовались Knowledge Graph Adapter и RAG Adapter с in-memory fallback. Точный статус внешнего graph backend и Graphiti/Graphify harness необходимо проверять по репозиторию.

В начале KG/RAG-задачи выполни поиск:

```powershell
rg -n -i "graphiti|graphify|graphifi|harness|knowledge.?graph|mcp" .
rg --files | rg -i "graph|rag|harness|mcp|eval"
```

Если harness существует, установи entrypoint, команду запуска, fixtures/seed, env names без секретов, внешний backend, fallback, eval-пороги и использование в CI. Проверь детерминизм, отсутствие смешения типов ОПО, корректные equipment/hazards/scenarios/services/appendices/regulations, fallback, отказ graph backend и provenance.

Если внешний граф является production-источником KG, рабочий harness обязателен для KG-backed приёмки. Если MVP официально использует in-memory fallback, harness не блокирует первый DOCX/PDF пилот, но должен быть закрыт до включения внешнего графа в production.

Не выдумывай путь или название harness, если они не найдены.

## 12. Шаблон DOCX и плейсхолдеры

`files/pmla_v2_template.docx` — критический бинарный артефакт.

Перед его изменением:

1. Зафиксируй текущий hash/размер.
2. Составь список изменяемых плейсхолдеров.
3. Проверь, не разбиты ли теги на несколько Word runs.
4. Не пересоздавай шаблон с нуля без явной необходимости.
5. Сохрани стили, media, section settings, headers/footers, TOC и PAGE fields.

После изменения:

- проверь целостность ZIP/DOCX;
- проверь отсутствие незаменённых тегов;
- проверь отсутствие raw Markdown и mojibake;
- сгенерируй документ на фиксированном fixture;
- отрендери и просмотри все страницы;
- проверь DOCX и PDF отдельно;
- зафиксируй новую версию/hash шаблона.

Должности и ФИО, а также номер и дата договора должны храниться отдельными полями, если это предусмотрено моделью и шаблоном.

## 13. Реальные данные

- Не коммить исходные документы с персональными данными, телефонами, подписями и реквизитами без явного решения пользователя.
- Для автотестов создавай обезличенный fixture с той же структурой и граничными случаями.
- Не выводи реальные секреты и персональные данные в логи и отчёты.
- Проверяй, что исходная анкета и `_import_meta` не встроились целиком в итоговый документ.
- Не придумывай отсутствующие ФИО, номера договоров, телефоны, расчётные показатели и нормативные реквизиты.
- Для каждого реального пилота сохраняй таблицу «источник → ожидаемое значение → context → место в DOCX».

## 14. Стратегия тестирования

Сначала найди реальные команды проекта. Не копируй команды из этого файла, если структура репозитория изменилась.

Порядок проверки изменения:

1. Самый узкий релевантный unit test.
2. Тесты изменённого сервиса/модуля.
3. Интеграционные тесты затронутого API/pipeline.
4. Полный backend test suite.
5. Frontend lint/test/build, если затронут frontend или API contract.
6. Runtime DOCX/PDF smoke, если затронуты context, schema, rules, renderer или template.
7. Quality Review и визуальный контроль для пользовательского документа.

Разделяй два E2E уровня:

### CI E2E

- API, БД, импорт, привязка и pipeline;
- обезличенный fixture;
- управляемый mock тяжёлого рендера/LibreOffice при необходимости;
- стабильный запуск в CI.

### Runtime E2E

- настоящий `pmla_v2_template.docx`;
- настоящий renderer;
- LibreOffice;
- реальные DOCX/PDF артефакты;
- постраничная и содержательная проверка.

Не объявляй E2E пройденным, если фактически проверен только один слой.

## 15. Проверка DOCX/PDF

Автоматически проверь открытие и целостность, незаменённые теги, Markdown/mojibake, обязательные разделы, строки таблиц, styles, TOC/PAGE, media и извлекаемую кириллицу PDF.

Визуально проверь титульный лист, длинные названия, ФИО/должности, разрывы таблиц, пустые заголовки/страницы, оглавление, номера страниц, приложения, подписи и различия Word/LibreOffice.

## 16. Классификация дефектов

Классифицируй замечания как `Import`, `Binding`, `Data`, `Mapper`, `Rules/Scenario`, `KG/RAG`, `Template`, `DOCX layout`, `PDF conversion`, `Regulatory/content`, `API/UX` или `Infrastructure/CI`.

Исправляй первопричину в соответствующем слое. Не используй ручную правку итогового DOCX как основное решение.

## 17. Изменения API, схемы и БД

- Сохраняй обратную совместимость, если пользователь не утвердил breaking change.
- Для изменения API обновляй типы frontend, validators, tests и документацию контракта.
- Для изменения schema добавляй позитивные и негативные fixtures.
- Для изменения модели БД создавай миграцию и проверяй upgrade path.
- Не редактируй уже применённую миграцию без доказанной необходимости; создавай следующую.
- Проверяй ошибки 404/422 отдельно от 500.
- Ошибка должна быть понятной и не оставлять частично сохранённые данные.

## 18. Git, commit и push

- Не создавай commit, не выполняй push и не открывай PR без явного запроса пользователя.
- Перед предложением коммита покажи:
  - выполненные тесты и их результат;
  - `git diff --stat`;
  - `git status --short`;
  - список известных ограничений.
- В commit включай только файлы текущей задачи.
- Не добавляй сгенерированные временные документы, логи и реальные данные.
- Не изменяй `.gitignore` только для сокрытия неожиданного мусора; сначала найди источник.
- Не переписывай историю Git без явного разрешения.

## 19. Формат итогового отчёта

Итог должен содержать:

1. Что получилось.
2. Какие файлы изменены.
3. Какие проверки выполнены и с каким результатом.
4. Какие блокеры или риски остались.
5. Один рекомендуемый следующий шаг.

Не скрывай `pending`, `skipped`, warnings и непроверенные внешние зависимости. Не называй задачу полностью завершённой, если runtime DOCX/PDF или ручная экспертиза ещё не выполнялись.

## 20. Definition of Done для пилота ПМЛА v2

Пилот завершён только когда импорт и binding воспроизводимы; анкета/context/schema сверены; обязательные данные дошли до DOCX; нет тегов и межтиповых утечек; DOCX/PDF прошли техническую и визуальную проверку; Quality Review содержит 0 critical; ручная сверка пройдена; повторная генерация эквивалентна; обезличенный fixture и разделённые CI/runtime E2E существуют; ограничения записаны в `HANDOFF.md`.

## 21. Вложенные инструкции

Если backend, frontend или каталог шаблонов требуют специальных правил, создавай более узкие файлы рядом с соответствующим кодом, например:

```text
AGENTS.md
backend/AGENTS.md
frontend/AGENTS.md
files/AGENTS.override.md
```

Вложенный файл должен содержать только отличия и специализированные команды. Не дублируй весь корневой документ. Перед созданием нового instruction-файла проверь фактическую структуру репозитория.
