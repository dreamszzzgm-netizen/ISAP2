# Задача: завершить превращение ПМЛА в шаблон docxtpl

## Контекст

Ты работаешь над Industrial Safety AI Platform (ISAP) — платформой автоматизации промышленной безопасности (см. `PROJECT_CHARTER.md`). Один из модулей — генерация документов через шаблоны Word (`docxtpl` + Jinja2), а не программную сборку `python-docx` с нуля.

Уже сделан MVP-шаблон: `pmla_v1_template.docx` + `pmla_v1.schema.json`. В нём:
- титульный лист и Таблица 1 (сведения об организации/ОПО) полностью параметризованы;
- сквозная замена названия организации, рег. номера, адреса, класса опасности, ФИО, ПАСФ по всему 79-страничному документу;
- Таблица 2 (оборудование) превращена в цикл Jinja2 (`{%tr for %}` / `{%tr endfor %}` — теги обязательно в ОТДЕЛЬНЫХ строках-маркерах, не внутри повторяемой строки).

**Не сделано:** таблицы 4, 6, 7, 8, 9, 10, 11 (сценарии аварий, травматизм, резерв ресурсов, дислокация сил) — данные в них статичны.

## Твоя задача

Довести шаблон до полной параметризации всех оставшихся таблиц с данными и подготовить сопутствующий код генератора.

## Обязательные skills (читать перед началом работы)

- `docx` — работа с .docx (структура OOXML, стили, таблицы, merge_runs). Прочитать SKILL.md перед любой правкой XML.
- `file-reading` / `pdf-reading` — если потребуется сверка с исходными эталонными документами в других форматах.
- Если создаёшь Python-модуль генератора — стандартный процесс создания файлов (см. правила по расположению файлов в /mnt/user-data/outputs и т.д., если работаешь в среде с этими путями).

## Субагенты (разбей работу на параллельные независимые потоки)

Каждый субагент получает **свою** таблицу и работает изолированно, чтобы не создавать конфликты правок в одном XML-файле. Финальную сборку (мердж изменений в один document.xml) делает координирующий агент последним шагом.

1. **`agent-table4-scenarios`**
   Таблица 4/6 — сценарии наиболее вероятных и опасных аварий. Самая объёмная и специфичная под объект. Превратить строки сценариев (С-1, С-2, ...) в цикл `{%tr for scenario in scenarios %}` с полями: `name`, `source`, `preconditions`, `signs`, `damaging_factors`.

2. **`agent-table7-8-incidents`**
   Таблицы 7 (травматизм) и 8 (происшедшие аварии) — статистические таблицы истории инцидентов. Цикл по годам/событиям: `{%tr for incident in incident_history %}`.

3. **`agent-table9-10-resources`**
   Таблица 9/10 — резерв материальных ресурсов (СИЗ, инструмент, оборудование — уже частично видели структуру). Цикл `{%tr for item in material_reserve %}` с полями `name`, `quantity`, `location`.

4. **`agent-table11-forces`**
   Таблица 11 — состав сил и средств, дислокация. Цикл `{%tr for force in engaged_forces %}` с полями `name`, `purpose`/`dislocation`.

5. **`agent-schema-merge`** (запускается последним, после всех остальных)
   - Собирает изменения всех агентов в единый `document.xml`.
   - Обновляет `pmla_v1.schema.json`, добавляя `required_tables` для каждой новой таблицы.
   - Прогоняет валидацию: `xmllint --noout document.xml`, затем тестовый рендер через `docxtpl` с синтетическим контекстом (минимум 2-3 элемента в каждом списке, чтобы проверить, что цикл действительно повторяет строки, а не ломает документ).
   - Проверяет, что после рендера не осталось необработанных `{{ }}` / `{% %}` меток (grep по итоговому document.xml).

## Обязательный процесс проверки (для каждого субагента)

1. Найти таблицу через `python-docx` по количеству строк/столбцов или уникальному тексту заголовка — не полагаться на порядковый номер таблицы в документе (могут сбиться при параллельных правках).
2. Взять ОДНУ строку с данными как образец цикла, остальные однотипные строки-примеры — удалить.
3. Тег `{%tr for %}` и `{%tr endfor %}` разместить в **отдельных строках-клонах** (склонировать XML строки данных, очистить текст, вставить только тег в первую ячейку) — см. готовый рабочий пример в `pmla_v1_template.docx`, таблица 2.
4. После правки — обязательно тестовый рендер через `docxtpl` с 2+ элементами в списке, чтобы убедиться, что Jinja не выдаёт `TemplateSyntaxError`.
5. Не трогать статический regulatory-текст (определения, ссылки на законы, глоссарий) — параметризовать только данные, специфичные для конкретного ОПО/организации.

## Формат сдачи

Каждый субагент возвращает:
- diff/патч своей таблицы в document.xml (или отдельный docx с изменением только своей таблицы);
- фрагмент `schema.json` для своей таблицы;
- лог успешного тестового рендера.

Координирующий агент — финальный собранный `pmla_v2_template.docx` + полный `pmla_v2.schema.json`.
