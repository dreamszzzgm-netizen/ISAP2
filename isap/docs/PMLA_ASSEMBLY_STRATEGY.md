# PMLA Assembly Strategy

**Версия:** 1.1
**Дата:** 2026-07-10

## Цель

Гибридная сборка ПМЛА: статичные разделы из шаблонов, переменные — с подстановкой данных, динамические — из анкеты/справочников/LLM, приложения — как файлы/манифест.

**Зачем:** снизить сложность генерации, сохранить форматирование, не заставлять LLM генерировать статичный контент.

## Что изменилось в v1.1

Реестр `ASSEMBLY_REGISTRY` перестал быть мёртвой классификацией — теперь он управляет DOCX-сборкой:

1. **Word TOC работает.** Заголовки разделов (level ≥ 1) получают встроенные стили `Heading 1`/`Heading 2`, а сами стили приведены к Times New Roman / чёрному цвету (`configure_heading_styles`). Теперь при «Обновить поле» в Word оглавление заполняется реальными разделами и подразделами сценариев. Ранее TOC был пустым — field вставлялся, но ни один заголовок не имел стиля Heading.
2. **Front matter по section_id, а не по русским строкам.** `_build_docx` убирает титульный лист, согласование, журнал корректировки и содержание из общего цикла секций через `get_front_matter_section_ids()` + `get_section_title()`, а не через хардкод `sections.pop("Журнал корректировки документа")`.
3. **Манифест приложений наполняется.** `appendices_manifest` синтезируется в `_enrich_context` из 5 канонических приложений реестра + статуса наличия из `attachments_checklist` анкеты (сопоставление по ключевым словам). Ранее поле всегда было пустым.

**Скоуп v1.1 — DOCX-сборка.** Engine routing (`*_SECTIONS` sets в каждом из 6 движков) и `EngineRouter` **не затронуты** — они работают и были опущены намеренно, чтобы не рисковать работающим пайплайном.

## Типы блоков

| Тип | LLM | Рендеринг | Когда использовать |
|-----|-----|-----------|-------------------|
| `static_block` | Нет | Фиксированный текст/таблица, без переменных | Аббревиатуры, термины, bibliography, журнал корректировки |
| `variable_block` | Нет | Jinja2-шаблон + подстановка данных из БД/анкеты | Титульный лист, согласование, характеристика ОПО, силы, оповещение |
| `generated_block` | Да | Шаблон + LLM/RAG для описательного текста | Введение, сценарии, взаимодействие, готовность, действия |
| `word_toc_block` | Нет | Word TOC field с Heading стилями | Содержание |
| `appendix_reference` | Нет | Ссылка на файл в манифесте приложений | Приложения 1-5 |
| `external_file` | Нет | Загруженный файл (будущее) | — |

## Классификация разделов ПМЛА

### static_block (4 раздела)

Разделы с фиксированным содержимым, не зависящим от данных объекта или организации. Не требуют ни переменных, ни LLM.

| Раздел | Описание | Почему static |
|--------|----------|---------------|
| `correction_log` | Журнал корректировки документа | Пустая таблица для заполнения вручную |
| `abbreviations` | Обозначения и сокращения | Стандартный набор: АСФ, ОПО, ПМЛА и т.д. |
| `terms` | Термины и определения | 22 термина из ФЗ-116, не меняются |
| `bibliography` | Список нормативных документов | 22 документа, фиксированный список |

**Импликации:**これらの разделы не передаются в Jinja2-рендерер и не вызывают LLM. Контент встраивается напрямую в DOCX.

### variable_block (11 разделов)

Разделы, которые рендерятся через Jinja2-шаблон с подстановкой данных из контекста (организация, объект, оборудование, вещества, лица). LLM не участвует.

| Раздел | Источник данных |
|--------|-----------------|
| `title_page` | organization, facility |
| `approval_sheet` | responsible_persons, approver |
| `section_1` | organization, facility, substances, equipment |
| `section_3` | facility (incidents) |
| `section_4` | emergency_services, forces |
| `section_6` | facility, organization (composition) |
| `section_8` | notification_scheme, contacts |
| `section_13` | material_reserve, insurance |
| `familiarization_sheet` | responsible_persons |
| `appendix_1` | organization |
| `appendix_5` | organization, facility, responsible_persons |

**Импликации:** LLM-провайдер не обязателен. Если LLM недоступен, эти разделы всё равно генерируются корректно.

### generated_block (10 разделов)

Разделы, требующие LLM для генерации описательного текста. Могут комбинировать FACT-слоты (код) + TEXT-слоты (LLM).

| Раздел | LLM-зависимость |
|--------|-----------------|
| `introduction` | LLM генерирует нормативную базу |
| `section_2` | LLM описывает сценарии + расчёты |
| `section_5` | LLM описывает взаимодействие |
| `section_7` | LLM описывает готовность |
| `section_9` | LLM описывает обмен информацией |
| `section_10` | LLM описывает первоочередные действия |
| `section_11` | LLM описывает действия персонала |
| `section_12` | LLM описывает безопасность населения |
| `special_section` | LLM + сценарии для оперативной части |

**Импликации:** При недоступности LLM используются fallback-тексты с пометкой "[Требуется уточнение]".

### word_toc_block (1 раздел)

| Раздел | Описание |
|--------|----------|
| `toc` | Содержание документа |

**Импликации:** В DOCX вставляется Word TOC field с инструкцией `TOC \o "1-2"`. Заголовки разделов (level 1) и подразделов сценариев (level 2) получают встроенные стили `Heading 1`/`Heading 2` (см. `configure_heading_styles`). Сами стили приведены к Times New Roman / чёрному цвету — визуал эталона сохраняется. Пользователь обновляет содержание в Word через ПКМ → Обновить поле. Заголовок «Содержание» рендерится без стиля Heading (level=0), иначе оглавление сошлось бы само на себя.

### appendix_reference (5 разделов)

Манифест приложений синтезируется из реестра (`get_appendix_manifest_entries`) + статуса наличия из `attachments_checklist` анкеты (сопоставление по ключевым словам в `_synthesize_appendices_manifest`). Реальные файлы приложений пока не загружаются (`external_file` — будущее).

| Раздел | Ключевые слова для match | Описание |
|--------|--------------------------|----------|
| `appendix_1` | изучен, обучен | Порядок изучения ПМЛА |
| `appendix_2` | оперативн, сообщен | Форма оперативного сообщения |
| `appendix_3` | пасф, состав формирован | Состав ПАСФ |
| `appendix_4` | оснащен, средств защит, сиз | Оснащение ПАСФ |
| `appendix_5` | оповещ, схема оповещ | Схема оповещения |

**Импликации:** В DOCX выводится таблица манифеста: № приложения, Наименование, Файл, Наличие. Поле «Файл» всегда `—` (загрузка файлов не реализована). Поле «Наличие» — `представлен`/`не представлен` в зависимости от совпадения с `attachments_checklist`.

## Почему содержание не надо считать вручную

Содержание (оглавление) строится автоматически из заголовков разделов. Каждый заголовок в DOCX использует стиль Heading 1 или Heading 2. Word может обновить содержание через встроенную функцию TOC. Ручной подсчёт страниц избыточен и создаёт риск рассинхронизации при изменении документа.

---

## Почему приложения лучше хранить как файлы/attachments

1. **Версионирование:** файлы приложений имеют свою историю изменений
2. **Форматирование:** сложные таблицы, схемы, графики лучше делать в Excel/Visio, чем генерировать через шаблон
3. **Переиспользование:** один файл приложения может использоваться в разных ПМЛА
4. **Юридическая сила:** приложения могут быть подписаны отдельно
5. **Размер:** встроенные изображения/файлы увеличивают DOCX

## Архитектура сборки

```
Questionnaire Data
       ↓
DocumentContext (from_dict)
       ↓
EngineRouter.generate_all()
  ├── static_block    → прямая вставка (без рендеринга)
  ├── variable_block  → Jinja2 template + context
  ├── generated_block → LLM + fallback
  ├── word_toc_block  → Word TOC field
  └── appendix_ref    → манифест таблица
       ↓
_build_docx()
  ├── Front matter: title_page, approval_sheet, correction_log, toc
  ├── Main sections: section_1..section_13, special_section
  ├── Appendices: appendix manifest table
  └── Back matter: bibliography, familiarization_sheet
```

## Реестр блоков

Единый источник истины: `backend/src/application/services/pmla_assembly_blocks.py`

```python
from src.application.services.pmla_assembly_blocks import (
    ASSEMBLY_REGISTRY,
    get_block_type,
    get_section_title,
    get_front_matter_section_ids,
    get_appendix_manifest_entries,
    get_static_sections,
    get_variable_sections,
    get_generated_sections,
)
```

Каждая запись `SectionBlockDef` несёт: `section_id`, `block_type`, `requires_llm`, `title` (русское название, синхронное со structure.json), `template`, `description`.

**Инвариант:** поле `title` в каждой записи реестра обязано совпадать с `title` в `structure.json` для того же `id`. Это защищает от рассинхронизации двух источников и проверяется тестом `test_registry_titles_match_structure_json`.

`structure.json` сохраняет существующие `content_type`/`slot_type` для обратной совместимости. `block_type` — ортогональная классификация.

## Что НЕ делает реестр (намеренно)

- **Не управляет engine routing.** Выбор движка (Template/Data/Scenario/Rules/Narrative) определяется хардкоженными `*_SECTIONS` sets в каждом engine и порядком в `EngineRouter`. `block_type` и эти sets — разные оси: почти совпадают, но не 1:1 (например `correction_log` = static, но рендерится TemplateEngine). Замена routing на registry — отдельная задача с высоким риском.
- **Не наполняет `corrections`.** Журнал корректировки заполняется вручную с юридическим основанием (`Основание`) и подписью (`Подпись`), которых нет в системе. Пустой дефолт (header + пустая строка) корректен для свежего документа.
- **Не загружает файлы приложений.** Блок `external_file` — заготовка для будущего функционала загрузки реальных файлов приложений.
