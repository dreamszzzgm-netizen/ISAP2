## 1. Обезличивание данных

Перед занесением реальных данных в систему их необходимо обезличить. Цель — убрать прямую идентификацию, сохранив структуру и инженерную значимость.

### 1.1 Что обезличивать

| Поле | Правило |
|---|---|
| ИНН / ОГРН | Заменить на заведомо условные (например `0000000000`, `0000000000000`). |
| Наименование организации | Заменить на условное (например `ООО «Демо Промбезопасность»`). |
| ФИО физических лиц | Заменить на роль/инициалы (например `Иванов И.И.` → «Ответственный за ПМЛА»). |
| Точные адреса | Указать населённый пункт и условный адрес (`г. Тюмень, ул. Учебная, д. 1`). |
| Телефоны | Заменить на шаблонные (`+7 (XXX) XXX-XX-XX`) или служебные номера для тестирования. |
| Email | Заменить на тестовые домены (`@example.com`). |
| Номера договоров/свидетельств | Заменить на условные (`А00-00000-0001`). |
| Заводские номера оборудования | Заменить на условные коды. |

### 1.2 Что НЕ обезличивать (инженерная значимость)

- Класс опасности ОПО.
- Тип ОПО (газопотребление / ХОПО / котельная и т.д.).
- Опасные вещества и их количества (кг).
- Физико-химические параметры (энергия взрыва, теплота сгорания, ПДК).
- Сценарии аварий и их последовательности.
- Состав сил и средств (типы, но не персональные данные).

### 1.3 Контроль обезличивания

Перед генерацией DOCX убедитесь, что в Листе сбора:
- [ ] Нет реальных ИНН/ОГРН.
- [ ] Нет реальных ФИО (только роли/инициалы).
- [ ] Нет точных адресов, позволяющих идентифицировать объект.
- [ ] Номера документов заменены на условные.

> Если используется реальный (не обезличенный) ОПО по согласованию — храните Лист сбора вне git и не коммитьте его.

## 7. Генерация DOCX

**UI:** карточка ОПО → анкета ПМЛА → вкладка «Генерация» → «Сгенерировать DOCX».

Или через API:

```powershell
curl -X POST `
  "http://localhost:8000/api/v1/pmla-questionnaires/<questionnaire_id>/generate" `
  -H "X-API-Key: isap-secret-2026" `
  -H "Content-Type: application/json" `
  -d '{}'
```

**Что происходит:** сборка контекста анкеты → 6 движков генерации → 29 разделов ПМЛА → quality review → запись DOCX.

Зафиксируйте `document_id` из ответа.

### 7.1 Что проверить сразу после генерации

- `status` = `pending_review` (ожидает проверки).
- `version` = `1`.
- `context_quality.passed` = `true` (иначе — доработать анкету).
- `quality_review.score` и `quality_review.overall_status` (`ok` / `warning` / `critical`).

## 8. Проверка quality review

**UI:** карточка ОПО → документ ПМЛА → блок «Quality Review».

Quality review содержит 10 проверок (`pmla_quality_review_service.py`):

| Код проверки | Что проверяет | Статусы |
|---|---|---|
| `incident_history` | Заполнен блок сведений об авариях | ok / warning |
| `selected_scenarios` | Выбраны сценарии аварий | ok / warning |
| `pasf` | Привязана ПАСФ/АСФ | ok / warning |
| `emergency_services` | Привязаны аварийные службы | ok / warning / critical |
| `organization_resources` | Описаны силы и средства | ok / warning |
| `notification_scheme` | Заполнена схема оповещения | ok / warning / critical |
| `financial_reserve` | Указан финансовый резерв | ok / warning |
| `insurance` | Указано страхование | ok / warning |
| `attachments_checklist` | Отмечены приложения | ok / warning |
| `docx_created` | DOCX успешно создан | ok / critical |

**Действия по статусам:**
- `critical` → **обязательно** устранить перед скачиванием (чаще всего — недостающие ПАСФ/службы/схема оповещения или DOCX не создан).
- `warning` → допустимо, но фиксируется как замечание для инженера.
- `ok` → блок в порядке.

Запишите итоговый `score` и все `warning`/`critical` — они пойдут в отчёт валидации.

## 10. Manual review workflow

**UI:** документ ПМЛА → блок «Проверка документа» (review workflow).

Workflow поддерживает переходы (поле `allowed_transitions`):

| Переход | Смысл |
|---|---|
| `in_review` | Взять на проверку |
| `needs_changes` | Отправить на исправление (есть замечания) |
| `approved` | Отметить как проверенный |
| `ready_to_issue` | Подготовить к выдаче |
| `issued` | Отметить как выданный |
| `archived` | Архивировать |

### 10.1 Прохождение

1. **`in_review`** — инженер берёт документ на проверку.
2. Открывает DOCX и проходит чеклист `PMLA_ENGINEER_REVIEW_CHECKLIST.md`.
3. Если есть замечания → **`needs_changes`**, вносит комментарий.
4. Замечания устраняются (доработка анкеты → регенерация, либо ручная правка DOCX инженером).
5. Повторная проверка → **`approved`**.
6. **`ready_to_issue`** → **`issued`** (по согласованию).

> Автоматическое «approved» недопустимо. Статус `approved` ставит только инженер после прохождения чеклиста.

## 11. Фиксация замечаний инженера

Все замечания фиксируются централизованно, чтобы собрать доказательную базу по качеству MVP.

### 11.1 Где фиксировать

Создайте файл (не коммитить без проверки):

```text
docs/inbox/PMLA_REAL_OPO_<CODE>_FINDINGS.md
```

Структура файла замечаний:

```markdown
# PMLA <CODE> — Engineer Findings

**ОПО:** <обезличенное наименование>
**document_id:** <id>
**Дата проверки:** <YYYY-MM-DD>
**Инженер:** <ФИО/роль>
**Quality review score:** <score> / overall: <status>

## Замечания

| # | Раздел ПМЛА | Тип | Замечание | Действие | Статус |
|---|---|---|---|---|---|
| 1 | Общие сведения об ОПО | содержание | ... | доработать анкету | open |
| 2 | (любое) | формат | сырой JSON в тексте | баг/доработка | open |

## Вывод по пригодности DOCX как инженерного черновика

<одно-два предложения: пригоден / требует доработки / непригоден>
```

### 11.2 Типы замечаний

- `содержание` — некорректные/неполные данные.
- `формат` — пустые поля, `None`/`undefined`, сырой JSON, HTML-теги.
- `структура` — несоответствие структуре ПМЛА (отсутствующие разделы).
- `нормативное` — несоответствие требованиям (требует инженерной экспертизы).

### 11.3 Где заполнять статус документа

В UI review workflow статус документа (`needs_changes` / `approved` и т.д.) выбирается из `allowed_transitions`. Комментарий инженера вводится в поле «Комментарий к проверке».

## 12. Что НЕ делать на этом этапе

- Не добавлять RAG, геокодинг, маршруты — это вне scope.
- Не коммитить Лист сбора, замечания и скачанный DOCX без проверки.
- Не ставить статус `approved` без прохождения чеклиста.
- Не пытаться «залатать» замечания кодом — этап собирает замечания, а не вносит правки.
