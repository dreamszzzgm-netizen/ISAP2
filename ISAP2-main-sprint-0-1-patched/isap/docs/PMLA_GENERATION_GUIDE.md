## 1. Что такое ПМЛА

**ПМЛА** — План мероприятий по локализации и ликвидации последствий аварий на опасном производственном объекте (ОПО).

Разрабатывается в соответствии с:
- **ФЗ-116** «О промышленной безопасности опасных производственных объектов»
- **Постановление Правительства РФ №1437** от 15.09.2020
- **Приказ Ростехнадзора №531** от 15.12.2020
- **Приказ Ростехнадзора №472** от 11.12.2020

### Структура документа (27 разделов)

```
Титульный лист
Журнал корректировки
Содержание
Перечень обозначений и сокращений
Термины и определения
Введение
1. Характеристика объекта
2. Сценарии наиболее вероятных аварий
3. Характеристика аварийности
4. Количество необходимых сил и средств
5. Организация взаимодействия сил
6. Состав и дислокация сил
7. Порядок обеспечения готовности
8. Организация управления, связи и оповещения
9. Система обмена информацией
10. Первоочередные действия при аварии
11. Действия персонала при угрозе и возникновении аварий
12. Мероприятия по безопасности населения
13. Материально-техническое обеспечение
2. Специальный раздел (оперативная часть)
Приложение 1. Порядок изучения ПМЛА
Приложение 2. Форма оперативного сообщения
Приложение 3. Состав ПАСФ
Приложение 4. Оснащение ПАСФ
Приложение 5. Схема оповещения
Список литературы
Лист ознакомления
```

### 5.2 Детализация по шагам

#### Шаг 1: Сборка контекста

Из БД извлекаются:
1. `HazardousFacilityModel` по `facility_id`
2. `OrganizationModel` по `facility.organization_id`
3. `EquipmentModel[]` по `facility_id`
4. `HazardousSubstanceModel[]` по `facility_id`
5. `ResponsiblePersonModel[]` по `organization_id`

Если есть координаты (latitude/longitude), вызывается `EmergencyServiceFinder.find_all_nearest()` для поиска ближайших служб:

| Служба | Источник |
|--------|----------|
| Пожарная охрана | Геокодер + справочник ПЧ |
| Аварийно-спасательная служба | Геокодер |
| Медицинская помощь | Геокодер |
| Аварийная газовая служба | Геокодер |

#### Шаг 2: Расчётный блок

Для каждого вещества с `hazard_properties`:

| Расчёт | Метод | Входные данные | Формула | Выход |
|--------|-------|----------------|---------|-------|
| Зона взрыва | `tnt_equivalent_v1` | quantity_kg, explosion_energy_mj | R = K × (E/4.184)^(1/3) | Радиус поражения (м) |
| Тепловое излучение | `thermal_radiation_v1` | quantity_kg, combustion_energy_mj_kg | q(R) = Q/(4πR²) | Радиус термического поражения (м) |
| Токсическое поражение | `toxic_dispersion_v1` | quantity_kg, mac_mg_m3 | Гауссова модель дисперсии | Радиус токсического воздействия (м) |

#### Шаг 5: Генерация разделов

**TemplateEngine** (10 разделов) — чистые Jinja2-шаблоны:
```
title_page → sections/00_title_page.j2
correction_log → sections/00_correction_log.j2
toc → sections/00_toc.j2
abbreviations → sections/00_abbreviations.j2
terms → sections/00_terms.j2
appendix_1 → sections/30_appendix_1.j2
appendix_2 → sections/31_appendix_2.j2
appendix_5 → sections/34_appendix_5.j2
bibliography → sections/40_bibliography.j2
familiarization_sheet → sections/41_familiarization_sheet.j2
```

**DataEngine** (8 разделов) — данные из БД + TableEngine:
```
section_1 → Таблицы 1-3 (ОПО, оборудование, вещества)
section_3 → Таблицы 7-9 (аварийность, аналогичные объекты)
section_4 → Таблица 10 (силы и средства)
section_6 → Таблицы 11-13 (состав, дислокация, маршруты)
section_8 → Таблица 14 (оповещение)
section_13 → Материальное обеспечение
appendix_3 → Состав ПАСФ
appendix_4 → Оснащение ПАСФ
```

**ScenarioEngine** (2 раздела) — JSON-шаблоны сценариев:
```
section_2 → Таблицы 4-6 (матрица, источники, сценарии) + блок-схемы
special_section → Таблица 15 (действия по каждому сценарию)
```

**RulesEngine** (6 разделов) — параметризованные правила:
```
section_5 → Взаимодействие сил
section_7 → Готовность сил
section_9 → Обмен информацией
section_10 → Первоочередные действия
section_11 → Действия персонала
section_12 → Безопасность населения
```

**NarrativeEngine** (1 раздел) — LLM:
```
introduction → Введение (цели, задачи, нормативная база)
```

## 6. Движки генерации

### 6.1 Архитектура

```
EngineRouter
├── TemplateEngine    (10 sections, 0% AI)
├── DataEngine        (8 sections, 0% AI)
├── ScenarioEngine    (2 sections, 0% AI)
├── RulesEngine       (6 sections, 0% AI)
└── NarrativeEngine   (1 section, ~10% AI)
```

### 6.2 DocumentContext

Единый контекст, передаваемый между движками:

```python
DocumentContext(
    organization: dict,       # Название, ИНН, адрес
    facility: dict,           # Тип, класс опасности, координаты
    equipment: list[dict],    # Оборудование
    substances: list[dict],   # Вещества + hazard_properties
    persons: list[dict],      # Ответственные лица
    calculation_results: list[dict],  # Результаты расчётов
    scenarios: list[dict],    # Сценарии из матрицы
    year: int,                # Текущий год
    approver: dict,           # Для титульного листа
    personnel: list[dict],    # Для раздела 11
    facility_coords: dict,    # Широта/долгота
    emergency_services: list[dict],  # Ближайшие службы
    accident_samples: list[dict]|None,  # Примеры аварий
)
```

### 6.3 TableEngine (утилита)

Используется DataEngine и ScenarioEngine для рендеринга таблиц:

| Метод | Формат | Пример |
|-------|--------|--------|
| `render_pipe_table(headers, rows)` | Markdown pipe | `| № | Наименование |` |

## 8. Расчётный блок

### 8.1 Взрывная волна (TNT-эквивалент)

**Метод:** `tnt_equivalent_v1`
**Источник:** РД 03-409-01

```python
# Входные данные
substance_name = "Природный газ"
quantity_kg = 500
explosion_energy_mj = 50
physical_state = "газ"

# Расчёт
w_tnt = explosion_energy_mj / 4.184  # Мдж → тонн ТНТ
radius = K * (w_tnt ** (1/3))        # K = 1.0 для открытой площадки

# Результат
results = {
    "zone_radius_m": round(radius, 1),
    "tnt_equivalent_kg": round(w_tnt * 1000, 1),
    "physical_state": physical_state
}
```

### 8.2 Тепловое излучение

**Метод:** `thermal_radiation_v1`
**Источник:** ГОСТ Р 12.3.047-98

```python
# Входные данные
substance_name = "Природный газ"
quantity_kg = 500
combustion_energy_mj_kg = 55

# Расчёт
total_energy_mj = quantity_kg * combustion_energy_mj_kg
# q(R) = Q / (4 * pi * R^2)
# Пороговое значение: 12.5 кВт/м² (термическое поражение)
radius = math.sqrt(total_energy_mj * 1000000 / (4 * math.pi * 12500))

# Результат
results = {
    "thermal_radius_m": round(radius, 1),
    "total_energy_mj": round(total_energy_mj, 1),
    "threshold_kw_m2": 12.5
}
```

### 8.3 Токсическое поражение

**Метод:** `toxic_dispersion_v1`
**Источник:** Методика ОМТ

```python
# Входные данные
substance_name = "Хлористый водород"
quantity_kg = 100
mac_mg_m3 = 5  # ПДК
lc50_mg_m3 = 1000  # LC50

# Расчёт (Гауссова модель)
# R = (2M / (π × a_y × a_z × MAC))^(1/(b_y + b_z + 1))
# a_y, a_z, b_y, b_z — коэффициенты дисперсии

# Результат
results = {
    "toxic_radius_m": round(radius, 1),
    "mac_mg_m3": mac_mg_m3,
    "lc50_mg_m3": lc50_mg_m3,
    "physical_state": "газ"
}
```

## 9. Сборка документа DOCX

### 9.1 Метод _build_docx()

```python
def _build_docx(self, title: str, sections: dict[str, str], metadata: dict) -> bytes:
    doc = DocxDocument()

    # 1. Заголовок (уровень 0, по центру)
    heading = doc.add_heading(title, level=0)
    heading.alignment = WD_ALIGN_PARAGRAPH.CENTER

    # 2. Метаданные
    doc.add_paragraph(f"Версия: {metadata['version']}")
    doc.add_paragraph(f"Дата генерации: {metadata['generated_at']}")
    doc.add_paragraph(f"Статус: {metadata['status']}")

    # 3. Разделы
    for section_title, content in sections.items():
        doc.add_heading(section_title, level=1)
        for line in content.strip().split("\n"):
            if line.strip():
                doc.add_paragraph(line.strip())

    # 4. Расчётные данные
    if metadata.get("calculation_results"):
        doc.add_heading("Расчётные данные", level=1)
        for calc in metadata["calculation_results"]:
            doc.add_paragraph(f"- {calc['method_id']}: {calc.get('substance', '—')}")

    # 5. Замечания валидации
    if metadata.get("validation_issues"):
        doc.add_heading("Результаты автоматической валидации", level=1)
        for issue in metadata["validation_issues"]:
            doc.add_paragraph(f"- [{issue['severity']}] {issue['section']}: {issue['reason']}")

    buffer = io.BytesIO()
    doc.save(buffer)
    return buffer.getvalue()
```

## 10. Валидация и AI-ревью

### 10.1 DocumentValidator (4 проверки)

| Проверка | Метод | Что проверяет | Severity |
|----------|-------|---------------|----------|
| Обязательные разделы | `_check_mandatory_sections` | «характеристика объекта», «сценарии аварий», «действия при аварий» | error/warning |
| Соответствие расчётов | `_check_numbers_match` | Числа из расчётов出现在 в тексте | warning |
| Наличие контактов | `_check_contacts` | Есть ли ответственные лица с телефонами | error/warning |
| Нормативные ссылки | `_check_regulatory_references` | Ссылки на НПА соответствуют действующим | warning |

**Валидация проходит, если нет ошибок severity="error".**

### 10.2 AIReviewer (11-пунктный чек-лист)

| # | Пункт | Что проверяет LLM |
|---|-------|-------------------|
| 1 | Сценарии соответствуют типу ОПО | Каждый сценарий соответствует типу и классу |
| 2 | Сценарии содержат причину, развитие, зоны | Причина + развитие + зоны поражения |
| 3 | Зоны поражения совпадают с расчётами | Описанные зоны соответствуют расчётам |
| 4 | Мероприятия привязаны к сценариям | Каждое мероприятие привязано к сценарию |
| 5 | СИЗ соответствуют классу опасности | Средства защиты соответствуют классу |
| 6 | Системы защиты учтены | Пожаротушение, оповещение, газоспасательная |
| 7 | Алгоритм действий логичен | Последовательность действий логична |
| 8 | Ответственные лица присутствуют | Все лица из данных присутствуют в документе |
| 9 | Порядок оповещения указан | Процедура оповещения (кто/когда/как) |
| 10 | Аварийные службы совпадают со справочником | Службы соответствуют ближайшим по координатам |
| 11 | Разделы не противоречат друг другу | Нет противоречий между разделами |

**Решения:**
- `auto_approve` — confidence ≥ 0.85 AND 0 failed checks
- `needs_revision` — >3 low-confidence items OR confidence < 0.5
- `escalate_to_human` — все остальные случаи

## 13. Жизненный цикл документа

```
┌─────────────┐
│ processing  │  ← Генерация
└──────┬──────┘
       │
       ▼
┌──────────────────┐
│ pending_review   │  ← Готов к ревью
└──────┬───────────┘
       │
       ├─── approved  ← Утверждён (review_date = now + 5 лет)
       │
       └─── rejected  ← Возвращён на доработку
                          │
                          ▼
                   ┌──────────────┐
                   │ processing   │  ← Перегенерация
                   └──────────────┘

┌──────────────────────────┐
│ auto_validation_failed   │  ← Автовалидация не прошла
└──────────────────────────┘
       │
       ▼
┌──────────────┐
│ processing   │  ← Исправление + перегенерация
└──────────────┘
```

### Сроки пересмотра

- При утверждении: `review_date = approved_at + 5 лет`
- `GET /api/v1/pmla/expiring?days=30` — документы в ближайшие 30 дней
- `GET /api/v1/pmla/overdue` — просроченные документы
