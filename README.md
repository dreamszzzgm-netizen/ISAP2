# ISAP2

**ISAP2** — информационная система для управления данными организаций и опасных производственных объектов, подготовки ПМЛА, учёта экспертиз промышленной безопасности, ведения CRM и интеллектуальной обработки технической документации с использованием AI, LLM и RAG.

Проект объединяет прикладные модули промышленной безопасности, документооборот, поиск по нормативно-технической базе, генерацию DOCX/PDF и AI-ассистента, работающего на основе проверяемых корпоративных источников.

> **Статус:** активная разработка. Отдельные функциональные направления могут находиться на разных стадиях готовности.

---

## Назначение системы

ISAP2 предназначена для централизации данных и процессов, связанных с промышленной безопасностью:

- ведение организаций и их реквизитов;
- ведение карточек опасных производственных объектов;
- хранение состава, классификации и характеристик ОПО;
- подготовка и генерация ПМЛА;
- учёт экспертиз промышленной безопасности;
- управление клиентами, обращениями, задачами и сделками;
- поиск по нормативной, технической и проектной документации;
- применение AI для анализа, классификации и подготовки документов.

- учёт экспертных организаций;
- учёт экспертов и ответственных специалистов;
- хранение номеров, дат, сроков действия и статусов заключений;
- контроль сроков и уведомления;
- хранение связанных документов и приложений;
- историю изменений и согласований;
- поиск и фильтрацию по объектам, организациям, статусам и срокам.

### CRM

Целевой CRM-модуль предусматривает:

- ведение клиентов и контрагентов;
- контактных лиц;
- обращений и запросов;
- сделок и этапов продаж;
- задач и напоминаний;
- истории звонков, писем и встреч;
- привязки коммуникаций к организациям, ОПО, экспертизам и проектам;

- ответственных сотрудников;
- отчётность по воронке и активности;

## AI, LLM и RAG

ISAP2 использует AI-компоненты для поиска, анализа и подготовки технической документации.

### Назначение AI-компонентов

- семантический поиск по документам;
- поиск по нормативно-технической базе;
- классификация загруженных файлов;
- извлечение фактов и реквизитов;
- преобразование неструктурированных документов в структурированные данные;
- подготовка черновиков технических документов;
- формирование ответов с опорой на найденные источники;
- сопоставление требований нормативных документов с объектами и данными системы;
- выявление противоречий и отсутствующих сведений;
- интеллектуальная помощь при заполнении анкет и карточек.

### RAG-пайплайн

```text
Документы и нормативная база
        ↓
Извлечение и нормализация текста
        ↓
Разбиение на фрагменты
        ↓
Построение embeddings
        ↓
ChromaDB и Knowledge Graph
        ↓
Семантический поиск
        ↓
Формирование контекста
        ↓
LLM
        ↓
Ответ, анализ или проект документа
```

### Основные принципы

- LLM не используется как единственный источник фактов;
- ответы формируются на основе найденных документов и данных ISAP2;
- источники должны сохраняться и отображаться пользователю;
- права доступа учитываются при поиске информации;
- секреты и персональные данные не передаются во внешние модели без разрешения;
- качество поиска и ответов контролируется автоматизированными тестами;
- критически значимые технические решения требуют проверки специалистом;
- AI-результат рассматривается как вспомогательный материал, а не как автоматическое экспертное заключение.

## Knowledge Graph

Knowledge Graph используется для связывания сущностей предметной области:

```text
Организация
    ├── ОПО
    │   ├── состав
    │   ├── здания и сооружения
    │   ├── технические устройства
    │   ├── опасные вещества
    │   ├── лицензии
    │   ├── ПМЛА
    │   └── экспертизы промышленной безопасности
    ├── контакты
    ├── обращения
    ├── сделки
    └── документы

Нормативный документ
    ├── требования
    ├── область применения
    ├── связанные объекты
    └── подтверждающие документы
```

Графовая модель дополняет векторный поиск и помогает учитывать связи между организациями, объектами, оборудованием, документами и нормативными требованиями.

## Архитектура

```text
Frontend
Next.js + TypeScript
        ↓
Backend API
FastAPI + Python
        ↓
PostgreSQL ─── ChromaDB
        │          │
        │          └── embeddings / semantic search
        │
        ├── Organizations
        ├── Hazardous Facilities
        ├── PMLA
        ├── Industrial Safety Expertise
        ├── CRM
        ├── Documents
        └── Knowledge Graph / RAG
```

### Пайплайн генерации ПМЛА

```text
Questionnaire
        ↓
Context Mapper
        ↓
PmlaOoxmlFlatRenderer
        ↓
DOCX Template
        ↓
DOCX
        ↓
PDF
```

Активный renderer:

```text
PmlaOoxmlFlatRenderer
```

Активный шаблон:

```text
isap/files/pmla_v2_template.docx
```

Renderer поддерживает:

- split-run placeholders;
- dotted tokens;
- циклы и строки таблиц;
- XML-экранирование;
- сохранение изображений и media;
- сохранение нетекстовых OOXML-частей;
- фиксацию `generation_pipeline`.

## Миграции базы данных

Изменения схемы выполняются через Alembic:

```powershell
docker compose exec backend alembic upgrade head
```

Ad-hoc скрипты с ручным `DROP COLUMN` и другими деструктивными операциями не должны заменять production-миграции.

Перед применением миграций к важным данным рекомендуется создавать резервную копию базы.

## Работа с Git

Рекомендуемый порядок:

```powershell
git status --short --branch
git diff --check
git add -- "точный/путь/к/файлу"
git diff --cached --check
git commit -m "тип(область): описание"
git push origin HEAD
```

Рекомендуемые типы commit:

```text
feat      новая функциональность
fix       исправление
test      тесты
docs      документация
refactor  изменение структуры без изменения поведения
chore     техническое обслуживание
ci        изменения CI
```

Не рекомендуется использовать `git add .` без предварительной проверки рабочего дерева.

## Что нельзя коммитить

- пароли, ключи и токены;
- `.env` с реальными учётными данными;
- абсолютные пользовательские пути;
- `node_modules`;
- `.next`;
- `__pycache__`;
- локальные E2E-артефакты;
- временные DOCX и PDF;
- журналы запуска;
- дампы с персональными или промышленными данными;
- ad-hoc destructive SQL-скрипты вместо Alembic-миграций.

## Безопасность

- Секреты передаются через переменные окружения.
- Секреты не должны выводиться в логи.
- Все входные файлы должны проходить проверку.
- Доступ к документам должен учитывать роли и права пользователей.
- AI-компоненты не должны обходить модель доступа ISAP2.
- Перед commit необходимо проверять staged diff.
- Перед миграциями необходимо создавать резервные копии важных данных.
- Локальные runtime-артефакты не должны попадать в Git.

## Ограничение ответственности
